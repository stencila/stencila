# Content Credentials Interoperability Testing

This directory contains a small evidence harness for C2PA conformance prep. The
goal is to show that Stencila can generate manifests other C2PA implementations
can read, and that Stencila can validate manifests generated by other
implementations.

The C2PA Conformance Program recommends interoperability testing before
submission. Treat each run as an evidence packet: keep the commands, tool
versions, verifier outputs, and asset digests together.

## Tools

Required:

- Stencila CLI from this workspace: `cargo run --bin stencila -- ...`
- `c2patool`, the CAI command-line implementation

Install `c2patool` from the official docs:

- <https://opensource.contentauthenticity.org/docs/c2patool/>

Optional manual check:

- Content Credentials Verify: <https://contentcredentials.org/verify>

## Environment

For local development signing:

```sh
cargo run --bin stencila -- credentials init
```

For conformance or production signing, provide real signing material instead of
the dev cert:

```sh
export STENCILA_CREDENTIALS_CERT=/path/to/cert-or-chain.pem
export STENCILA_CREDENTIALS_KEY=/path/to/private-key.pem
export STENCILA_CREDENTIALS_TSA_URL=https://tsa.example.org
```

For local signer-trust checks, Stencila uses the official C2PA trust-list cache
by default. Refresh it explicitly before an evidence run when you want to record
the exact trust-list material used:

```sh
cargo run --bin stencila -- credentials trust refresh
cargo run --bin stencila -- credentials trust status --as json
```

Use `STENCILA_CREDENTIALS_TRUST_ANCHORS` only when you need to override the
official cache with a local PEM bundle. Trust will only become `yes` if the
asset's signing chain traces to one of the loaded anchors.

You can also pass the TSA URL directly:

```sh
cargo run --bin stencila -- credentials sign image.png --tsa-url https://tsa.example.org
```

## Test 1: Stencila Generator, c2patool Validator

This signs a PNG with Stencila, then asks both Stencila and `c2patool` to read
the result.

```sh
rust/content-credentials/interop/stencila-to-c2patool.sh
```

Default output:

```text
target/content-credentials-interop/stencila-to-c2patool/
```

Evidence produced:

- `tool-versions.txt`
- `digests.txt`
- `stencila-sign.txt`
- `stencila-verify.txt`
- `stencila-verify.json`
- `stencila-inspect.json`
- `c2patool-summary.json`
- `c2patool-detailed.json`

If `C2PATOOL_TRUST_ANCHORS` is set, the script also writes:

- `c2patool-trust.json`

Manual browser check:

1. Upload `target/content-credentials-interop/stencila-to-c2patool/stencila-signed.png` to <https://contentcredentials.org/verify>.
2. With the dev cert, expect the issuer to be unrecognized.
3. With conformance signing material, expect the issuer result to reflect the public trust-list status of that certificate.

## Test 2: Other Generator, Stencila Validator

Use an asset signed by another implementation, such as `c2patool`, Adobe
Content Authenticity, OpenAI Media Service, or camera output.

```sh
rust/content-credentials/interop/other-to-stencila.sh path/to/signed-asset.jpg
```

Default output:

```text
target/content-credentials-interop/other-to-stencila/
```

Evidence produced:

- `tool-versions.txt`
- `digests.txt`
- `stencila-verify.txt`
- `stencila-verify.json`
- `stencila-inspect.json`
- `c2patool-summary.json`
- `c2patool-detailed.json`

If `C2PATOOL_TRUST_ANCHORS` is set, the script also writes:

- `c2patool-trust.json`

If the asset uses a sidecar manifest, keep the `.c2pa` file next to the asset.
The scripts follow the CAI sidecar convention, for example `photo.jpg` and
`photo.c2pa`.

## Suggested Matrix

Run the generator and validator tests across:

| Format | Expected Stencila signing mode |
| --- | --- |
| PNG | Embedded |
| JPEG | Embedded |
| WebP | Embedded |
| SVG | Embedded |
| PDF | Sidecar |

For the conformance application, record at least:

- Stencila version or commit SHA.
- `c2patool` version.
- C2PA SDK versions reported in manifests.
- Operating system.
- Signing certificate profile used: dev, internal, or C2PA trust-list cert.
- Whether timestamping was used.
- Whether public Verify recognized the issuer.

## Current Known Limits

- Local dev certificates are intentionally untrusted by public verifiers.
- Public trust-list recognition requires a suitable C2PA signing certificate.
- Stencila currently verifies trust according to the local SDK configuration; it
  does not yet ship a polished trust-list management command.
- The custom `org.stencila.provenance` assertion is still minimal and intended
  to grow in the next implementation slice.
